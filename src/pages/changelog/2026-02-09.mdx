### Tasks Done & Summary
- Added **ShareOfVoiceParamsContext** (shareOfVoice-params.context.tsx): startDate, endDate, filtersAnd.
- **ShareOfVoiceRoute** provides that context (dates from computeDates, filters from **buildAndFromSelected**(search params)).
- **useShareOfVoiceDates** and **useShareOfVoiceFilters** use context when present, otherwise store (hooks always called, no conditional hook).
- SOV selectors now use **buildAndFromSelected** (same as Discovery) instead of key.toLowerCase().
#### **Why we added the context and provider**
- On the **TanStack** SOV route, filters and dates live in the **URL** (search params). We never write them into the Zustand params store.
- The **subscription** (useShareOfVoiceFilterSubscriptions) and **metrics table** (useGetShareOfVoiceMetricsTableData) were still reading from the **store** via useShareOfVoiceDates() and useShareOfVoiceFilters(), so they kept seeing default/stale values and didn’t react to filter/date changes.
- We needed a way for those hooks to get the **current URL-derived** dates and filters when we’re on the TanStack route, without changing how the React Router SOV page works (it still uses the store).
- So we added **ShareOfVoiceParamsContext** and **ShareOfVoiceParamsProvider**: the TanStack **ShareOfVoiceRoute** computes startDate, endDate, and filtersAnd from the URL and provides them via context. **useShareOfVoiceDates** and **useShareOfVoiceFilters** were updated to use that context when present and fall back to the store otherwise. That way the subscription and table get the right values on the TanStack route, and the React Router SOV page is unchanged. Same idea as Discovery’s context: one place provides the “current” params (from URL on TanStack, from store on React Router).
