- **Analysis filter totals & date range** – Sidebar filter counts on the Analysis route now use `useAnalysisParamsForQueries`, so they stay in sync when the date range (or other Analysis URL params) changes.
- **Sidebar dates on Analysis** – Removed the extra `useNewsFeedDates()` call from `useAnalysisParamsForQueries` so the sidebar doesn’t rely on the news-feed params store on the Analysis (TanStack Router) route; normalised `feedId` to `undefined` when the context returns `null`.
- **useNewsFeedSidebarFilters & useDynamicFilters** – Fixed TypeScript spread/object-type issues in the query variables (ternaries and `as Record<string, unknown>` where needed) and fixed the ref type in useDynamicFilters (`feedId ?? null` for `lastClearedFeedIdRef`).
- **Effective dates** – Confirmed that `useAnalysisParamsForQueries` follows the same effective start/end logic as `useNewsFeedContext` (feed volume bar range when selected, otherwise route/filter range).
- **Unit tests** – Added **useAnalysisParamsForQueries.test.ts** (both branches: with/without Analysis provider, feedVolume, null feedId). Updated **useDynamicFilters.test.ts** so the `useNewsFeedContext` mock includes `startDate`/`endDate` and removed the unused `useNewsFeedDates` mock. Updated **useNewsFeedSidebarFilters.test.ts** to mock `useAnalysisParamsForQueries` (dropped `useFeedParamsStore`/`useParams`), added a “pauses when params missing” test, and fixed the chartSelection tests’ mocks.
- **Nav v2 cleanup doc** – In **todo/navigation-v2-cleanup.md**: added files to delete (e.g. useUrlParamsRouter, useNewsFeedParamsRouter, url-params), clarified the “remove/refactor useNewsFeedContext” step (and its relation to `useAnalysisParamsForQueries`), added “remove store URL-sync methods” and `enable-discovery-tab-stable` to the checklist, and linked to the migration doc.
- **TanStack migration doc** – In **docs/tanstack-router-migration.md**: expanded “What Gets Deleted” with the full file list (with paths and reasons), code blocks to remove (including store URL-sync methods), feature flags to remove, and a pointer to the cleanup doc.
- **useGetFeedMetrics:** Switched from `useNewsFeedContext` to `useAnalysisParamsForQueries`, added `pause: !enabled` when `feedId` / `startDate` / `endDate` are missing. Updated **useGetFeedMetrics.test.ts** to mock `useAnalysisParamsForQueries` instead of `useNewsFeedContext` and added a test that the query is paused when required params are missing.
- **ArticleContentControlPanel:** Switched to `useAnalysisParamsForQueries` and added `queryEnabled` + `pause: !queryEnabled` for the aggregate articles query. No dedicated test file; news-feed and store tests confirmed.
- **Feed volume graph (`useNewsFeedVolumeGraphData.ts`):** Adjusted `filters.dateRange` so that when using a **preset range** (`input.daysSince` + `input.period`) the backend is not sent start/end in filters: `dateRange` is set to `\{ startDate: null, endDate: null \}`. When **custom range** (`isCustomDaysRange`), `dateRange` is sent with real `startDate`/`endDate`. Applied in both `useNewsFeedVolumeData` and `useNewsFeedVolumeChartData`.
- **useArticlesPagination:** Switched from `useNewsFeedContext` to `useAnalysisParamsForQueries`; existing guards still prevent fetch when params are missing.
- **useArticlesPagination.test.ts:** Added tests that mock `fetchArticlesPaged`, `useAnalysisParamsForQueries`, and `useInView`. Tests cover: return value (`loadMoreRef`, `pagination`, `isFetchingArticles`), initial fetch with correct args when params present, no fetch when `feedId` / `startDate` / `endDate` is undefined, `selectedFilters` passed through, and scroll container ref passed to `useInView` as `root`. Imports use `@/test-utils/test-utils` for `renderHook` and `waitFor`.
- **NewsFeedContent audit:** Reviewed children for analysis reuse. **NewsFeedContent** still uses `useNewsFeedContext` for search (needs switch to `useAnalysisParamsForQueries`). **NewsFeedControlPanel** uses `useNewsFeedParamsRouter` and store (tied to news-feed page); **AppliedFilters** and **NewsFeedControlsMoreOptions** are store/context-driven. **FeedMetrics**, **NewsFeedVolumeGraph**, **ArticleContentControlPanel**, **ArticlesContainer** already use params via `useAnalysisParamsForQueries`.
- **Deferred:** Updating **NewsFeedContent** to use `useAnalysisParamsForQueries` and hiding **NewsFeedControlPanel** (and thus **AppliedFilters**) when inside analysis context (e.g. when `useAnalysisParamsContext()` is non-null); to be done later.
